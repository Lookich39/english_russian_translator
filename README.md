# English → Russian Translation Pipeline

🏆 **Результат на соревновании Yandex Data Dojo 2026:** **35 место из 320 участников**  
📊 **Итоговая метрика:** **BLEU 27.7**

Это решение для задачи машинного перевода **английских абзацев на русский язык**.  
Проект выполнен **в рамках соревнования Yandex Data Dojo 2026** и реализует inference-pipeline для перевода длинных связных текстов с упором на:

- согласованность терминов,
- корректную передачу местоимений и референции,
- сохранение структуры абзацев,
- аккуратную обработку прямой речи, списков и технических фрагментов.

В основе решения — **google/gemma-4-E2B-it** и двухпроходная схема:
1. **Draft translation** — черновой перевод;
2. **Revision** — редактура черновика на основе исходного текста.

---

## Возможности

- Перевод **с английского на русский**.
- Поддержка **абзацного перевода**, а не только отдельных предложений.
- Двухпроходный inference:
  - черновой перевод,
  - последующая редактура.
- Более стабильная передача:
  - терминологии,
  - местоимений,
  - структуры текста,
  - цитат и прямой речи.
- Очистка результата от лишних служебных фраз и markdown-артефактов.
- Поддержка запуска в формате соревнования:
  - вход: `input.pickle`
  - выход: `output.json`
- Поддержка запуска через **Docker**.

## Архитектура решения

```text
input.pickle
     │
     ▼
┌──────────────────────────┐
│  Pass 1: Draft           │
│  GENERAL_PROMPT          │
│  gemma-4-E2B-it          │
└──────────────────────────┘
     │
     ▼
┌──────────────────────────┐
│  Pass 2: Revision        │
│  REVISION_PROMPT         │
│  gemma-4-E2B-it          │
└──────────────────────────┘
     │
     ▼
┌──────────────────────────┐
│  Postprocessing          │
│  очистка вывода          │
└──────────────────────────┘
     │
     ▼
output.json
```

### Почему два прохода?

При переводе длинных абзацев модель в один проход чаще ошибается в таких местах:

- теряет согласованность местоимений;
- по-разному переводит один и тот же термин;
- меняет стиль прямой речи;
- ломает структуру текста.

Второй проход позволяет исправить эти проблемы, используя уже готовый черновик перевода.

## Подготовка

На вход подаётся файл:

```text
input.pickle
```

Формат:

```python
[
    {
        "rid": int,
        "src": str,
    },
]
```

Где:
- `rid` — идентификатор примера,
- `src` — английский текст для перевода.

## Запуск через Docker

Сборка образа:

```bash
docker build -t my-solution .
```

Запуск на примере:

```bash
mkdir -p out

docker run --rm --gpus all \
  -v "$PWD/example-input.pickle":/workspace/input.pickle \
  -v "$PWD/out":/workspace/out \
  my-solution
```

Если GPU нет, можно убрать флаг:

```bash
--gpus all
```

Проверка результата:

```bash
cat out/output.json
```

## Основные идеи решения

### 1. Prompt-based translation
Для первого прохода используется специализированный prompt, который просит модель:

- точно сохранять смысл,
- одинаково переводить термины по всему абзацу,
- сохранять структуру текста,
- не добавлять комментариев и пояснений.

### 2. Revision pass
После получения чернового перевода выполняется второй проход, где модель редактирует перевод с учётом:

- исходного английского текста,
- уже сгенерированного русского черновика.

Это улучшает:
- связность,
- точность терминологии,
- передачу референции,
- итоговый BLEU.
